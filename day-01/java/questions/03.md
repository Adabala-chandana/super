# Q03 — ExecutorService: bounded async order processing

You run an order-fulfillment service. Incoming orders must be processed asynchronously, but you must not spawn unlimited threads under load.

## Scenario

- Orders arrive as a `List<Order>` (id + payload).
- Processing one order takes ~50–200ms of simulated work (`Thread.sleep` is fine).
- You must process them concurrently with a **fixed-size** pool.
- The API that accepts the batch must **wait until all orders finish** (success or failure) before returning a summary.
- If one order fails, others must still complete; failures are collected, not aborted silently.

## Requirements

1. Use `ExecutorService` with a fixed thread pool (e.g. size 4).
2. Submit one task per order; collect results with `Future` / `invokeAll` / `CompletableFuture` (your choice).
3. Return a summary: `{ processed: N, failed: M, failedOrderIds: [...] }`.
4. Always shut down the pool (or use try-with-resources / a shared pool with proper lifecycle) so the JVM can exit cleanly.

## Constraints

- Do **not** use `new Thread()` per order.
- Pool size must be bounded (no cached unbounded pool for this exercise).
- Caller blocks until the batch is done.

## Expected behavior

```text
Input:  10 orders, pool size 4, 2 orders throw on purpose
Output: processed=8, failed=2, failedOrderIds=[3, 7]
```
