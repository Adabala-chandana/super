# Q04 — ConcurrentHashMap: request-rate counters under load

An API gateway needs per-user request counts for the current minute so it can enforce a soft rate limit. Many request-handler threads update the same map at once.

## Scenario

- Key = `userId` (`String`), value = request count for the current window (`Integer` / `Long` / `AtomicInteger` — your choice).
- On each request: increment that user's count **atomically**.
- Occasionally a background task resets the whole window (clear or replace the map) without corrupting in-flight increments.
- You must support concurrent reads of a user's current count while updates happen.

## Requirements

1. Use `ConcurrentHashMap` (not a plain `HashMap` + manual locking for the hot path).
2. Implement:
   - `long recordRequest(String userId)` — increment and return the new count
   - `long getCount(String userId)` — current count or `0`
   - `void resetWindow()` — start a new window safely
3. Prefer atomic map operations (`compute`, `merge`, or `AtomicInteger` values) so two threads for the same user never lose an increment.

## Constraints

- Do not synchronize the entire map with `synchronized` on every request.
- `recordRequest` must be correct under concurrent calls for the **same** `userId`.
- After `resetWindow()`, new requests start counting from zero (eventual consistency of in-flight calls is acceptable if you document it).

## Expected behavior

```text
// 8 threads each call recordRequest("u1") 10_000 times
getCount("u1") == 80_000

resetWindow();
getCount("u1") == 0
```
