# Q05 — Shared resource: connection / permit pool

A service talks to a downstream system that allows at most `MAX_PERMITS` concurrent calls (e.g. DB connections, HTTP client slots, or licensed API calls). Worker threads must acquire a permit, use the resource, then release it — even when the work throws.

## Scenario

- `MAX_PERMITS = 3`.
- Up to 20 worker tasks try to call `downstream.call(payload)` at the same time.
- At any moment, at most 3 workers may be inside `call`.
- If no permit is free, a worker waits (optionally with timeout).
- On timeout or failure, the permit must still be released (no permanent leak).

## Requirements

1. Model the shared resource with something production-like:
   - `Semaphore` with `MAX_PERMITS`, **or**
   - a small fixed pool of `Connection`-like objects handed out / returned
2. Implement `Result invokeProtected(Callable<Result> work)` that:
   - acquires a permit / connection
   - runs `work`
   - always releases in `finally`
3. Drive it with an `ExecutorService` submitting ~20 tasks; assert (via logging or a counter) that concurrent in-flight calls never exceed `MAX_PERMITS`.

## Constraints

- Losing a permit on exception is considered a bug (simulate a leak detector: available permits back to `MAX_PERMITS` when idle).
- Do not use a single global `synchronized` lock around the entire downstream call if you use a `Semaphore`/`pool` (the point is bounded concurrency, not full serialization beyond the permit count).

## Expected behavior

```text
MAX_PERMITS=3, 20 tasks
peakInFlight <= 3
after all tasks finish: availablePermits == 3
```
